# Obsidian-Kanban Fork: Trello Sync Implementation Plan

This plan describes how to fork [obsidian-community/obsidian-kanban](https://github.com/obsidian-community/obsidian-kanban) and add two-way Trello sync. The result is a single plugin that **is** the kanban board AND syncs with Trello — no separate plugin needed.

---

## Architecture Overview

The forked plugin keeps 100% of obsidian-kanban's existing functionality. Trello sync is added as a **parallel layer** that:

1. **Pull**: Fetches Trello board data → converts it to the kanban Board data model → merges/replaces lanes and items → triggers `StateManager.saveToDisk()` which serializes to the kanban markdown format
2. **Push**: Intercepts board mutations (drag-drop moves, item creation, item edits, item deletion) → calls the Trello API to mirror each change
3. **Linking**: Each kanban board file can optionally be linked to a Trello board via frontmatter (`trello_board_id`, `trello_api_key`, `trello_token`). Non-linked boards work exactly as before.

### Key Principle: Work With the Existing Data Model

The kanban plugin already has Board → Lane → Item. Trello has Board → List → Card. The mapping is direct:
- Trello List ↔ Kanban Lane
- Trello Card ↔ Kanban Item
- Card description ↔ Linked note body (via wiki-link in item title)
- Card metadata (labels, due, members) ↔ Item metadata + linked note frontmatter

## Phase 3: Pull — Trello → Kanban Board

This is the most complex phase. The pull operation fetches all Trello data and reconciles it with the current kanban board state.

### Task 3.2: Implement `pull()` — full board sync

**Step-by-step algorithm:**

```
1. Read the kanban board's Trello frontmatter to get `trello.board_id`
2. Load the sidecar metadata (TrelloBoardMetadata)
3. Fetch from Trello API (all in parallel):
   - getLists(boardId)           → TrelloList[]
   - getCards(boardId)           → TrelloCard[] (with ?actions=commentCard)
   - getMembers(boardId)         → TrelloMember[]
4. Get the current board state from StateManager:
   - const board = stateManager.getBoard() 
   
5. RECONCILE LANES (Trello Lists → Kanban Lanes):
   For each Trello list (sorted by pos):
     a. Look up in metadata: does a kanbanLaneId exist for this trelloListId?
     b. YES — lane exists:
        - Find the lane in board.children by kanbanLaneId
        - If the Trello list name changed → update lane.data.title
        - Update pos ordering (reorder board.children to match Trello list order)
     c. NO — new list from Trello:
        - Create a new Lane using stateManager.newLane(trelloListName)
        - Insert it into board.children at the correct position
        - Add a new TrelloLaneMapping to metadata
   
   For each existing kanban lane that has a TrelloLaneMapping but whose trelloListId is NOT in the fetched lists:
     - The list was archived/deleted in Trello
     - Move the lane's items to the kanban archive (board.data.archive)
     - Remove the lane from board.children
     - Remove the TrelloLaneMapping from metadata

6. RECONCILE ITEMS (Trello Cards → Kanban Items):
   For each Trello card:
     a. Look up in metadata: does a kanbanItemId exist for this trelloCardId?
     b. YES — item exists:
        i.  Find which lane it's currently in (by kanbanItemId in lane.children)
        ii. Check if the card moved to a different list:
            - If card.idList !== metadata.trelloListId for this item's lane mapping
            → Remove item from old lane, insert into new lane
        iii. Check for content changes (card name):
            - If card.name !== item.data.titleRaw (after stripping wiki-link syntax)
            → Update item.data.titleRaw
        iv. Update position within lane (sort by card.pos)
        v.  Sync the linked note if it exists (see Task 3.3)
        vi. Sync comments via TrelloCommentStore
     c. NO — new card from Trello:
        - Create a new Item: the title is a wiki-link to a note file
          e.g., `[[card-name-a1b2|Card Name]]`
        - Create the linked note file with frontmatter (see Task 3.3)
        - Create the sidecar comment file
        - Insert the item into the correct lane at the correct position
        - Add a new TrelloItemMapping to metadata
   
   For each existing kanban item that has a TrelloItemMapping but whose trelloCardId is NOT in the fetched cards:
     - The card was archived/deleted in Trello
     - Move the item to board.data.archive
     - Move the linked note to .archived/ subfolder (or delete per settings)
     - Remove the TrelloItemMapping from metadata

7. APPLY the reconciled board:
   - Call stateManager methods to update the board state
   - stateManager.saveToDisk() serializes the board back to kanban markdown
   
8. UPDATE METADATA:
   - Set lastFullSync to now
   - Save the sidecar metadata file

9. SHOW NOTICE with summary
```

### Task 3.3: Linked note management — `TrelloNoteManager.ts`

Each Trello card maps to both a kanban item AND an optional linked markdown note. The note holds the card's description as its body and Trello metadata in frontmatter.

```typescript
// src/trello/TrelloNoteManager.ts

export class TrelloNoteManager {
    constructor(private app: App) {}
    
    // Get the folder where linked notes live (same folder as the kanban board file)
    getNoteFolder(kanbanFilePath: string): string;
    
    // Create a new linked note for a Trello card
    async createNote(card: TrelloCard, members: TrelloMember[], kanbanFilePath: string): Promise<string>;
    
    // Update an existing linked note with fresh Trello data
    // Returns conflict info if both local and remote changed
    async updateNote(notePath: string, card: TrelloCard, members: TrelloMember[]): Promise<NoteUpdateResult>;
    
    // Read the note body to get the current card description
    async readNoteBody(notePath: string): Promise<string>;
    
    // Archive a note (move to .archived/ subfolder or delete)
    async archiveNote(notePath: string, shouldDelete: boolean): Promise<void>;
    
    // Generate the sanitized filename for a card
    sanitizeFilename(cardName: string, cardId: string): string;
    
    // Build the wiki-link text for a kanban item title
    buildItemTitle(cardName: string, cardId: string): string;
    
    // Extract the note path from a kanban item's wiki-link title
    extractNotePath(itemTitle: string, kanbanFilePath: string): string | null;
    
    // Compute SHA-256 hash of a string
    hashBody(body: string): string;
}

interface NoteUpdateResult {
    status: 'updated' | 'no-change' | 'conflict' | 'created';
    conflict?: {
        localBody: string;
        remoteBody: string;
        localTitle: string;
        remoteTitle: string;
    };
}
```

**Note frontmatter format** (same as current plugin):
```yaml
---
trello_card_id: "abc123"
trello_card_name: "Fix login bug"
trello_list_id: "def456"
trello_board_id: "ghi789"
trello_last_synced: "2026-05-19T12:00:00Z"
trello_content_hash: "sha256:e3b0c44..."
trello_pos: 16384
members:
  - username: "johndoe"
    fullName: "John Doe"
labels:
  - name: "Bug"
    color: "red"
due: "2026-06-01"
attachments:
  - name: "mockup.png"
    url: "https://trello.com/..."
---

The actual card description goes here.
```

**`createNote()` implementation details:**
1. Generate filename: `sanitizeFilename(card.name, card.id)` → `fix-login-bug-a1b2.md`
2. Build frontmatter object with all Trello metadata
3. Body = `card.desc || ''`
4. Write the file using `app.vault.create(notePath, content)`
5. Return the note path

**`updateNote()` implementation details:**
1. Read the existing note and extract body + frontmatter
2. Compute `currentHash = hashBody(currentBody)`
3. Compare `currentHash` to stored `trello_content_hash` → detects local changes
4. Compare `card.dateLastActivity` to `trello_last_synced` → detects remote changes
5. If both changed → return conflict
6. If only remote changed → overwrite note with Trello data, update hash/timestamp
7. If only local changed → leave note alone (user hasn't pushed yet)
8. If neither changed → no-op

**`buildItemTitle()` implementation:**
```typescript
buildItemTitle(cardName: string, cardId: string): string {
    const basename = this.sanitizeFilename(cardName, cardId).replace('.md', '');
    return `[[${basename}|${cardName}]]`;
}
```

## Phase 4: Push — Kanban Board → Trello

Push operations are triggered by user actions in the kanban UI. Each mutation that changes the board should optionally (if the board is Trello-linked) also call the Trello API.

### Task 4.1: Identify all mutation points in obsidian-kanban

The key mutations happen in these locations. Each needs a Trello sync hook.

**Item moves (drag-and-drop):**
- File: `src/dnd/managers/ScrollManager.ts` or `src/DragDropApp.tsx`
- Function: `handleDragEnd(event: DragEndEvent)`
- What happens: `moveEntity(board, sourcePath, destPath)` is called, then the board state is saved

### Task 4.2: Implement the push methods in `TrelloSync.ts`

**`pushItemMove(stateManager, itemId, newLaneId, newIndex)`:**
```
1. Look up the item's trelloCardId from metadata
2. Look up the new lane's trelloListId from metadata
3. Calculate the new Trello `pos` value:
   - Get the items in the target lane, sorted by current pos
   - If inserting at index 0: pos = firstItem.pos / 2
   - If inserting at end: pos = lastItem.pos + 16384
   - If inserting between two items: pos = (prevItem.pos + nextItem.pos) / 2
4. Call trelloClient.moveCard(trelloCardId, trelloListId, pos)
5. Update the linked note's frontmatter: trello_list_id, trello_pos
6. Update the item mapping in metadata
```

**`pushItemCreate(stateManager, laneId, itemTitle)`:**
```
1. Look up the lane's trelloListId from metadata
2. Parse the item title to extract the card name (strip wiki-link syntax if present)
3. Call trelloClient.createCard(trelloListId, cardName, '')
4. Create a linked note file with the card's frontmatter
5. Update the kanban item's title to include the wiki-link: [[note-basename|Card Name]]
6. Add a new TrelloItemMapping to metadata
7. Create an empty sidecar comment file
8. Return the new item's kanbanItemId
```

**`pushItemUpdate(stateManager, itemId, newTitle)`:**
```
1. Look up the item's trelloCardId from metadata
2. Parse the new title to extract the card name
3. Call trelloClient.updateCard(trelloCardId, { name: newCardName })
4. If the linked note exists, update its frontmatter (trello_card_name)
5. If the card name changed enough to affect the filename, rename the note file
   using app.vault.rename() (which updates internal links)
```

**`pushItemDelete(stateManager, itemId)`:**
```
1. Look up the item's trelloCardId from metadata
2. Call trelloClient.deleteCard(trelloCardId)
   OR call trelloClient.updateCard(trelloCardId, { closed: true }) to archive instead
3. Archive/delete the linked note per user settings
4. Remove the TrelloItemMapping from metadata
```

**`pushLaneCreate(stateManager, laneName)`:**
```
1. Read the board's trello.board_id from frontmatter
2. Calculate pos: after the last existing list
3. Call trelloClient.createList(boardId, laneName, pos)
4. Add a new TrelloLaneMapping to metadata
5. Return the new lane's trelloListId
```

**`pushLaneRename(stateManager, laneId, newName)`:**
```
1. Look up the lane's trelloListId from metadata
2. Call trelloClient.updateList(trelloListId, { name: newName })
3. Update the TrelloLaneMapping name in metadata
```

**`pushLaneArchive(stateManager, laneId)`:**
```
1. Look up the lane's trelloListId from metadata
2. Call trelloClient.archiveList(trelloListId)
3. For each item in the lane: archive its linked note
4. Remove the lane and all its items from metadata
```

### Task 4.3: Wire push hooks into the kanban codebase

**Strategy**: Create a helper function that checks if the current board is Trello-linked and, if so, calls the appropriate push method. This minimizes changes to existing kanban code.

```typescript
// src/trello/TrelloHooks.ts

export function createTrelloHooks(plugin: KanbanPlugin) {
    return {
        // Returns true if this board has Trello sync enabled
        isTrelloBoard(stateManager: StateManager): boolean {
            const fm = stateManager.getBoard()?.data?.frontmatter;
            return !!fm?.trello?.board_id;
        },
        
        // Call after a successful drag-drop move
        async onItemMoved(stateManager: StateManager, itemId: string, newLaneId: string, newIndex: number): Promise<void>,
        
        // Call after a new item is created
        async onItemCreated(stateManager: StateManager, laneId: string, item: Item): Promise<void>,
        
        // Call after an item's content is updated
        async onItemUpdated(stateManager: StateManager, item: Item, oldTitle: string): Promise<void>,
        
        // Call before an item is deleted
        async onItemDeleted(stateManager: StateManager, itemId: string): Promise<void>,
        
        // Call after a new lane is created
        async onLaneCreated(stateManager: StateManager, lane: Lane): Promise<void>,
        
        // Call after a lane is renamed
        async onLaneRenamed(stateManager: StateManager, laneId: string, newName: string): Promise<void>,
        
        // Call before a lane is deleted/archived
        async onLaneArchived(stateManager: StateManager, laneId: string): Promise<void>,
    };
}
```

**Insertion pattern for each mutation point:**

At each mutation site identified in Task 4.1, add a call like:
```typescript
// EXISTING CODE:
const updatedBoard = moveEntity(board, sourcePath, destPath);
stateManager.setState(updatedBoard);
await stateManager.saveToDisk();

// ADD AFTER:
if (plugin.trelloHooks.isTrelloBoard(stateManager)) {
    await plugin.trelloHooks.onItemMoved(stateManager, itemId, newLaneId, newIndex)
        .catch(err => new Notice(`Trello sync failed: ${err.message}`));
}
```

Wrap every Trello hook call in `.catch()` so Trello API failures never break the kanban UI. The kanban board should always work offline — Trello sync is best-effort.

```
1. Collect all conflicts into an array

   a. Open ConflictModal for each conflict (or a batch modal)
```

## Phase 8: Card Metadata Display on Kanban Tiles

### Task 8.1: Enhance kanban card rendering with Trello metadata

The kanban plugin's `Item` component renders card titles. For Trello-linked items, enhance the rendering to show:
- **Label chips**: Colored dots/pills based on Trello labels
- **Due date badge**: With overdue styling
- **Member avatars/initials**: Small circles with first letters
- **Comment count badge**: Number of comments
- **Attachment icon**: If attachments exist

**Implementation approach:**

The kanban item title is a wiki-link like `[[card-a1b2|Card Name]]`. The linked note's frontmatter contains all the metadata (labels, due, members, etc.).

**Option A — Read from frontmatter at render time:**
- In the Item component, resolve the wiki-link to a file path
- Read the file's cached frontmatter via `app.metadataCache.getFileCache(file)?.frontmatter`
- Extract labels, due, members, attachments, comment count
- Render badges below the title

**Option B — Store metadata in kanban item data:**
- During pull, store card metadata in the Item's `data.metadata` field (kanban supports custom metadata)
- Read from `item.data.metadata` at render time
- Pro: No file I/O during render. Con: Metadata can go stale between syncs.

**Recommended: Option A** — it's always fresh and doesn't require changing the kanban data model. `metadataCache` reads are synchronous and fast.

## Phase 10: Testing and Edge Cases

### Task 10.1: Manual test matrix

Test each of these scenarios in both directions:

| Scenario | Direction | Expected |
|---|---|---|
| New card in Trello | Pull | Item appears in correct lane, note created |
| New card in Kanban | Create | Card created in Trello, note created locally |
| Card moved in Trello | Pull | Item moves to new lane |
| Card dragged in Kanban | Push | Card moves to new list in Trello |
| Card renamed in Trello | Pull | Item title updated, note file renamed |
| Card edited in Kanban | Push | Card name updated in Trello |
| Card deleted in Trello | Pull | Item archived, note archived |
| Card deleted in Kanban | Push | Card archived in Trello, note archived |
| List created in Trello | Pull | New lane appears |
| Lane created in Kanban | Push | New list created in Trello |
| List renamed in Trello | Pull | Lane title updated |
| Lane renamed in Kanban | Push | List renamed in Trello |
| Note body edited locally | Push note | Card desc updated in Trello |
| Card desc edited in Trello | Pull | Note body updated (or conflict if local edits) |
| Both local and remote edit | Pull | Conflict modal shown |
| Comment added locally | Push | Comment posted to Trello |
| Comment added in Trello | Pull | Comment appears in sidecar |
| Card reordered in Trello | Pull | Items reorder within lane |
| Card reordered in Kanban | Push | Card pos updated in Trello |
| Board with no Trello link | Any | Works exactly as vanilla obsidian-kanban |
| Invalid API key | Pull/Push | Friendly error Notice, no crash |
| Network offline | Push | Error Notice, kanban state preserved |
| Rate limit hit | Pull (large board) | Backoff + retry, progress notice |

### Task 10.2: Edge cases to handle

1. **Card with no description**: Note body is empty — valid
2. **Very long card name**: Filename truncated at 80 chars + hash
3. **Duplicate card names**: Hash suffix from card ID ensures uniqueness
4. **Special characters in card/list names**: `sanitizeFilename()` strips `\/\:*?"<>|`
5. **Cards moved to archived list in Trello**: Treated as deleted during pull
6. **Multiple boards**: Each kanban file links to one Trello board independently
7. **Stale metadata**: If sidecar JSON is deleted, pull rebuilds it from scratch
8. **Concurrent edits**: If two people edit via Trello while one edits locally, conflicts are caught on pull
9. **Empty board**: Zero lists/cards — renders empty kanban with no errors
10. **Board unlinked mid-session**: `isTrelloBoard()` returns false, hooks are no-ops
